1. General Concepts

A. Overview of Dynamic Web Application Components

  • Client-Side: HTML, CSS, JavaScript - handles presentation and user interaction
  • Server-Side: Java (Servlets/JSP), databases - handles business logic and data processing
  • Communication: HTTP/HTTPS protocols between client and server
  • Database: Persistent storage for application data (MySQL, PostgreSQL, etc.)

B. MVC Design Pattern

Model

Business logic and data

View

Presentation layer (JSP)

Controller

Handles user input (Servlet)

Separation of concerns: Divides application into three interconnected components to separate internal representations from how information is presented

Java component responsibilities:

  • Model: Java classes that represent data and business logic
  • View: JSP files that handle presentation
  • Controller: Servlets that manage application flow

3. Cloud Computing

A. Differences Between IaaS and PaaS

Feature IaaS (Infrastructure as a Service) PaaS (Platform as a Service)
Control Level High - manage servers, storage, networking Medium - focus on application development
Management Responsibility OS, middleware, runtime, data, applications Only data and applications
Scalability Manual scaling required Automatic scaling typically included
Example AWS EC2 AWS Elastic Beanstalk

B. AWS S3

  • Simple Storage Service (S3): Object storage service offering scalability, data availability, security, and performance
  • Use for Web Applications:
    • Host static websites (HTML, CSS, JavaScript, images)
    • Store and retrieve any amount of data at any time
    • Backup and archive critical data
    • Store assets like images, videos for web applications
  • Key Features: 99.999999999% durability, versioning, lifecycle policies, encryption

C. AWS EC2

  • Elastic Compute Cloud (EC2): Provides scalable computing capacity in the cloud
  • Deployment of Dynamic Applications:
    • Launch virtual servers (instances) with various OS options
    • Choose instance types based on CPU, memory, storage needs
    • Configure security groups to control traffic
    • Install required software (Java, Tomcat, MySQL, etc.)
    • Deploy application code to the instance
  • Key Features: Auto Scaling, Elastic Load Balancing, multiple instance types

D. AWS Elastic Beanstalk

  • What it is: PaaS that makes it easy to deploy and scale web applications
  • Difference from EC2:
    • EB handles deployment details automatically (capacity provisioning, load balancing, scaling)
    • EB manages the infrastructure while you focus on code
    • EB supports multiple platforms (Java, .NET, Node.js, etc.)
  • Key Terms: Application, Environment, Application Version, Environment Tier

E. AWS RDS

  • Relational Database Service (RDS): Managed relational database service
  • Use as Data Server:
    • Hosts the database in the cloud separate from application servers
    • Supports multiple database engines (MySQL, PostgreSQL, Oracle, etc.)
    • Automates administrative tasks (backups, patching, failure detection)
  • Benefits: Scalable, highly available, durable, secure

  • setStatus(code) - Set HTTP status code

5. Servlets

A. Adding a Servlet to a Dynamic Web Project

  1. Right-click project → New → Servlet
  2. Enter class name and package
  3. Configure URL mapping (e.g., "/login")
  4. Select methods to implement (doGet, doPost)
  5. Finish to create servlet class with annotations
URL Mapping:

Configured in web.xml or via annotations:

@WebServlet("/login")
public class LoginServlet extends HttpServlet {
    // Servlet code
}
doGet() vs. doPost():
  • doGet(): Handles HTTP GET requests (retrieving data)
  • doPost(): Handles HTTP POST requests (submitting data)
  • Typically call one from the other to avoid code duplication

B. Request and Response Objects in Servlet

protected void doPost(HttpServletRequest request, HttpServletResponse response) 
        throws ServletException, IOException {
    
    // Get form parameters
    String username = request.getParameter("username");
    String password = request.getParameter("password");
    
    // Process data (validate, authenticate, etc.)
    
    // Set attributes for JSP
    request.setAttribute("message", "Login successful");
    
    // Forward to JSP
    request.getRequestDispatcher("/result.jsp").forward(request, response);
}

C. Using Java Objects from Model in Servlet

protected void doPost(HttpServletRequest request, HttpServletResponse response) 
        throws ServletException, IOException {
    
    // Create or retrieve model object
    UserService userService = new UserService();
    
    // Process request using model
    User user = userService.authenticate(
        request.getParameter("username"),
        request.getParameter("password")
    );
    
    // Store model object for view
    request.setAttribute("user", user);
    
    // Forward to view (JSP)
    request.getRequestDispatcher("/profile.jsp").forward(request, response);
}

6. MVC Design Pattern

A. Why MVC?

Separation of Concerns:

  • Improves maintainability by separating application into distinct components
  • Easier to manage complexity as application grows
  • Allows parallel development (different team members can work on different components)
  • Promotes code reuse (models can be used with different views)
  • Easier to test components in isolation

B. Components of MVC

Model

  • Java classes representing data and business logic
  • Handles database operations
  • Contains validation rules
  • Independent of UI

View

  • JSP files for presentation
  • Displays data from model
  • Minimal business logic
  • Can include JSTL/EL for cleaner code

Controller

  • Servlets that handle user input
  • Processes requests and responses
  • Interacts with model
  • Selects appropriate view

Connections Between Components:

Model
View
Controller
Updates
Notifies
Uses
Updates
  • User interacts with view (clicks button, submits form)
  • View sends request to controller
  • Controller processes request, interacts with model
  • Model updates data and notifies controller
  • Controller selects view and passes updated data
  • View renders with updated data from model

Data Passing Between Components:

Controller to View:
  • request.setAttribute(name, value)
  • session.setAttribute(name, value) for session data
  • Accessed in JSP using EL: ${attributeName}
View to Controller:
  • Form parameters via request.getParameter(name)
  • URL parameters in GET requests
  • Hidden form fields for additional data

C. Security Aspects

Client-Side Validation

  • JavaScript validation for better user experience
  • Quick feedback without server round-trip
  • Limitation: Can be bypassed (always need server validation)
  • Example: Check for empty fields, valid email format

Server-Side Validation

  • Essential for security and data integrity
  • Validate all input from client
  • Check for required fields, data types, length, format
  • Example: Validate username/password in servlet

Preventing SQL Injection

  • Use PreparedStatement with parameterized queries
  • Never concatenate user input into SQL strings
  • Validate and sanitize all user input
  • Use principle of least privilege for database users
  • Example: SELECT * FROM users WHERE username = ?

D. JDBC Components

Connection Object

  • Manages connection to database
  • Obtained via DriverManager.getConnection()
  • Must be closed when done

PreparedStatement

  • Precompiled SQL statement
  • Prevents SQL injection
  • Parameters set with setXxx() methods

ResultSet / Int

  • ResultSet for SELECT queries (read operations)
  • int for INSERT, UPDATE, DELETE (CUD operations)
  • Represents number of affected rows

E. Preventing SQL Injection with PreparedStatement

What is SQL Injection?

A code injection technique where malicious SQL statements are inserted into an entry field for execution, potentially allowing attackers to view, modify, or delete database data.

How PreparedStatement Prevents It:

  • Uses parameterized queries where user input is treated as data, not executable code
  • Input values are properly escaped automatically
  • SQL structure is fixed at preparation time
Vulnerable Code (Concatenation):
String query = "SELECT * FROM users WHERE username = '" + username + 
               "' AND password = '" + password + "'";
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery(query);

Dangerous! Allows SQL injection.

Secure Code (PreparedStatement):
String query = "SELECT * FROM users WHERE username = ? AND password = ?";
PreparedStatement pstmt = conn.prepareStatement(query);
pstmt.setString(1, username);
pstmt.setString(2, password);
ResultSet rs = pstmt.executeQuery();

Secure! Prevents SQL injection.

F. General Steps for Working with DB

  1. Create connection object: 
    // Load driver (not needed for newer JDBC versions)
Class.forName("com.mysql.cj.jdbc.Driver");

// Create connection
Connection conn = DriverManager.getConnection(url, username, password);
  2. Create and set up PreparedStatement: 
    String sql = "INSERT INTO products (name, price) VALUES (?, ?)";
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, product.getName());
pstmt.setDouble(2, product.getPrice());
  3. Execute query/update: 
    // For SELECT queries
ResultSet rs = pstmt.executeQuery();

// For INSERT, UPDATE, DELETE
int rowsAffected = pstmt.executeUpdate();
  4. Deal with results: 
    // Process ResultSet
while (rs.next()) {
    String name = rs.getString("name");
    double price = rs.getDouble("price");
    // Create objects or process data
}

// Check rows affected for updates
if (rowsAffected > 0) {
    // Success
}
  5. Close connection: 
    // Close resources in reverse order
rs.close();
pstmt.close();
conn.close();

    Note: Use try-with-resources for automatic closure:

    try (Connection conn = DriverManager.getConnection(url, username, password);
     PreparedStatement pstmt = conn.prepareStatement(sql)) {
    // Work with database
} catch (SQLException e) {
    // Handle exceptions
}

8. Deploying Full-Stack, Java MVC Applications to the Cloud

A. Platform as a Service vs Infrastructure as a Service

PaaS (Platform as a Service)

  • Common Scenarios:
    • Web application hosting
    • API backend services
    • Development and testing environments
  • Advantages:
    • Simplified deployment process
    • Automatic scaling
    • Built-in load balancing
    • Managed infrastructure (no server maintenance)
    • Faster time to market
  • Example: AWS Elastic Beanstalk, Heroku, Google App Engine

IaaS (Infrastructure as a Service)

  • Common Scenarios:
    • Custom infrastructure needs
    • Legacy application hosting
    • Complete control over environment
  • Advantages:
    • Full control over virtual machines
    • Flexibility in configuration
    • Ability to install custom software
    • More granular security controls
  • Example: AWS EC2, Google Compute Engine, Microsoft Azure VMs

B. Elastic Beanstalk

What is Elastic Beanstalk?

AWS Elastic Beanstalk is an easy-to-use service for deploying and scaling web applications and services. You simply upload your code and Elastic Beanstalk automatically handles the deployment, from capacity provisioning, load balancing, auto-scaling to application health monitoring.

Elastic Beanstalk Terms

  • Application: Logical collection of Elastic Beanstalk components (environments, versions, configurations)
  • Application Version: Specific labeled iteration of deployable code
  • Environment: Version deployed to AWS resources (EC2 instances, etc.)
  • Environment Tier: Web server (handles HTTP requests) or Worker (handles background tasks)

Difference Between Elastic Beanstalk and Manual EC2

Feature Elastic Beanstalk Manual EC2
Setup Complexity Simple - just upload code Complex - configure everything
Scaling Automatic Manual configuration
Load Balancing Built-in Manual setup
Monitoring Basic monitoring included Manual setup
Control Limited - AWS manages infrastructure Full control

C. Deploying to Elastic Beanstalk

What is a Java WAR file?

A Web Application Resource (WAR) file is a JAR file used to distribute a collection of JavaServer Pages, servlets, Java classes, XML files, static web pages, and other resources that together constitute a web application.

  • Standard format for packaging Java web applications
  • Contains all components needed for deployment
  • Created by Eclipse when exporting a Dynamic Web Project
  • Deployed to application servers like Tomcat

Deployment Steps

  1. Create WAR file:
    • In Eclipse: Right-click project → Export → WAR file
    • Or use Maven/Gradle build tools
  2. AWS Console:
    • Navigate to Elastic Beanstalk service
    • Click "Create Application"
